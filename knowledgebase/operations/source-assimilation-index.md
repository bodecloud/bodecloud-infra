# Source Assimilation Index

This page is the retrieval contract for the knowledgebase.

It exists because `bolabaden-infra` has exactly the kind of source sprawl that
produces documentation which sounds polished, exhaustive, and still misses the
point:

- live Compose runtime
- architecture-intent instruction surfaces
- planning docs with stronger language than current proof supports
- orchestration side-paths that matter without being present truth
- archive pressure showing what the user is actually sick of

If those all get flattened into one confident voice, the docs become smooth,
generic, and dishonest.

The user is not mainly asking for more words.
They are asking for documentation that can still distinguish:

- dream
- runtime
- intent
- promotion work
- exploration
- non-proof

That distinction is what "actually RAG this time" means here.

The user is also asking for something harsher than ordinary retrieval quality:

> reconstruct the real dream from scattered, uneven, partially contradictory
> artifacts without flattening it into a simpler adjacent dream

That is why this page cannot be satisfied by citation density.
The retrieval problem here is not "find many relevant files."
It is "recover the real architecture pressure without allowing the easiest file
to define the whole question."

This page matters because a repo like this can easily produce the illusion of
deep understanding through sheer document volume.

The user is explicitly rejecting that illusion.
They want the retrieval process to rebuild the actual architecture pressure,
not merely decorate it with many citations.

This page therefore should not be read as an index in the ordinary sense.
It is closer to a quality bar for reconstruction.
Its job is to stop the knowledgebase from sounding well-sourced while still
quietly retrieving a smaller, safer, more ordinary version of the user's real
question.

They are also rejecting a calmer but equally dangerous illusion:

- the page cites the right files
- the citations sound balanced
- balance starts sounding like the system is already well-understood
- the hidden operator reconstruction burden disappears into a smooth summary

That last step is where "good retrieval" quietly turns back into polished
ambiguity.

The sharper version is:

- more citations are not necessarily more truth
- more balance is not necessarily more fidelity
- more composure is not necessarily more understanding

In this repo, those can all be symptoms of the same failure: the system has
remembered the right materials while forgetting what the user was actually
forcing those materials to answer.

## What "actually RAG this time" means in this repo

In this repo, real retrieval does **not** mean:

- mention more filenames
- quote more text
- add more background
- sound more academic

It means every serious claim must do all of the following:

1. identify the exact class of claim being made
2. go to the highest-authority source for that class first
3. say what that source proves
4. say what that source does **not** prove
5. preserve contradictions instead of smoothing them away
6. let the current worktree outrank elegant prose for runtime claims
7. let the archive restore the user's real pressure without pretending the
   archive is runtime evidence

If a rewrite skips those steps, it can become longer and still fail the user's
standard completely.

It also has to do one more thing:

8. resist the temptation to answer the most normal version of the question when
   the user is clearly asking a stranger and more demanding one

That eighth rule is what separates actual assimilation from polished
near-miss documentation.

There is a ninth rule implied by the rest:

9. say whether an apparent success is explained by system-owned truth or by
   human reconstruction that the system still quietly depends on

Without that rule, a retrieval pass can correctly cite the runtime, the
planning docs, and the archive and still miss the sharpest thing the user keeps
asking for.

There is also an implied tenth rule:

10. keep the repo's negative benchmark alive, so the docs do not become
    satisfied by an answer that would sound reasonable to generic self-hosting
    culture but still feel insulting to this user

That is what keeps retrieval here from collapsing into a broad but ordinary
"infrastructure understanding" pass.

## The real question the retrieval stack is serving

The docs are not ultimately trying to answer:

- how is the repo organized?
- which orchestrator is best?
- which reverse proxy should I use?

They are trying to answer this:

> can several ordinary Docker nodes be made to behave like one
> request-preserving, operator-readable personal cloud without forcing the user
> into either brittle static glue or immediate heavyweight orchestrator
> surrender?

That is the only question that explains:

- why the archive matters
- why wrong-node behavior matters
- why stateful honesty matters
- why "just use Kubernetes" keeps reading as a non-answer

If a retrieval pass forgets that question, it starts collecting surface facts
instead of reconstructing the actual problem.

The important word there is "reconstructing."

And reconstruction here has to stay adversarial.

The pass is not done when it can tell a coherent story.
It is done when it can say which parts of that story are owned by the runtime
itself and which parts still live in the operator's head.

That difference is the actual retrieval target.
The repo is not only trying to retrieve facts.
It is trying to retrieve the current boundary between shared truth and private
operator folklore.

This page also implies a held-out test for the knowledgebase as a whole:

can a reader leave the site able to distinguish:

- what is live
- what is intended
- what is exploratory
- what is still fake-option theater

without privately reconstructing that map themselves?

If not, the assimilation pass has still failed, even if it cited many files.

That held-out test matters more than page-level elegance.
If the reader still has to mentally fuse several registers of truth after
reading the site, then the site has only relocated the reconstruction burden,
not removed it.

The real system the user wants is only partially visible in any one artifact.
Some files show the live runtime.
Some show a clearer dream than the runtime implements.
Some show rejected futures.
Some show pressure and frustration more honestly than the polished docs do.

Real retrieval in this repo therefore behaves less like "find the answer" and
more like "assemble the missing architecture shape from uneven evidence without
pretending the gaps are closed."

## The assimilation failure this page is designed to stop

The knowledgebase regresses whenever it quietly merges these into one register:

- live root-runtime truth
- repo-native architecture intent
- planning and future-promotion truth
- exploratory branch truth
- archive refusal truth
- generic infra common sense imported from outside the repo

Each of those answers a different question.
None of them are allowed to impersonate the others.

That sentence should be treated as a hard anti-flattening rule.
Most of the worst documentation failures in this repo happen when one of those
registers borrows the confidence or emotional weight of another.

This is the main anti-fog rule for the whole documentation set.

It is also why top-level pages cannot be excused for being "just summaries."
In this repo, summary pages are some of the highest-risk pages, because they
are exactly where truth layers get merged into one calmer voice.

That means summary quality in this repo has to be judged differently from
normal docs.

A summary is not better just because it is:

- smoother
- more compact
- more evenly weighted
- easier to read without discomfort

If those gains come from deleting ownership ambiguity or deleting the remaining
reconstruction tax, the summary has become less truthful.

This is why summary work in this repo should feel slightly over-corrective.
The docs are not merely fighting clutter.
They are fighting a repeated tendency to sound complete by quietly removing the
very discomfort that tells the truth.

## The user's hidden demand this index must preserve

The user is not simply asking for accuracy at the sentence level.
They are asking the docs to stop quietly shrinking their real demand into
easier adjacent questions.

The hidden demand is:

- do not swap "make any-node entry work honestly" for "document the proxy"
- do not swap "remove sacred-node memory" for "list all services"
- do not swap "preserve request meaning under failure" for "describe the
  topology"
- do not swap "find the smallest honest middle layer" for "compare
  orchestrators broadly"

If a retrieval pass returns a technically correct but smaller question, it is
still a bad retrieval pass.

That is the most common near-miss this knowledgebase has to keep preventing.
The repo already has enough material to answer many adjacent questions well.
The difficulty is refusing to confuse that with answering the exact one the
user actually cares about.

That is the central trap in this repo.

## What this page is and is not allowed to prove

This page is authoritative about:

- how retrieval should be routed
- how source classes should be ordered
- how to detect assimilation failure

It is not authoritative about:

- whether a given runtime claim is true
- whether a future option has earned promotion
- whether archive pressure has already become implementation

This page governs method, not outcome.

Another way to say the same thing is:

the docs fail whenever they become satisfied with an answer the wider
self-hosting ecosystem would consider reasonable, while the user is still
clearly asking for the thing those reasonable answers never actually close.

It is always easier to retrieve:

- what services exist
- what proxies are present
- what orchestration tools were explored
- what plans sound mature

than to retrieve:

- what specific kind of hidden sacred-node knowledge the user is trying to
  eliminate
- what exact wrong-node behavior is still unproven
- what kind of "option" is actually fake because it leaves the same operator
  burden intact

That asymmetry is the real reason this page exists.
It is easier to inventory, easier to compare, easier to summarize, and easier
to narrate respectably than it is to preserve the user's real standard for
what would count as relief.

That asymmetry has to stay visible because it explains why so much seemingly
competent documentation still misses the point.

The easy material is usually:

- inventories
- plans
- components
- respectable future directions

The hard material is usually:

- where the system still relies on socially reconstructed truth
- where failure handling is still narratively stronger than operationally owned
- where a stronger control layer would still leave the user asking the same
  question in slightly fancier terms

## Assimilation failure checks

Before accepting a rewritten page as "well assimilated," ask:

1. Did the page answer the real question, or a safer neighboring question?
2. Did runtime claims start from runtime artifacts, or from cleaner prose?
3. Did intent claims start from the strongest intent witness?
4. Did planning text get treated as if naming a repair means the repair is
   nearly owned?
5. Did archive pressure restore the user's standard without being mistaken for
   shipped proof?
6. Did the page preserve contradiction, or did it purchase coherence by
   shrinking the dream?
7. Can a reader leave the page knowing what still depends on private operator
   reconstruction?

If the answer to any of those is no, the assimilation pass is still incomplete.

This tier is the one most often over-read.
In this repo, "the component exists" is usually just the start of the real

It is also where many false-success summaries get born.

Once the component is present, it becomes tempting to tell a story in which the
remaining burden must already be mostly procedural.

The user is explicitly asking the docs not to make that leap.

That is why retrieval here must be adversarial toward convenience.

If the easiest artifact says "there is a helper" and the harder artifact says
"the helper still deletes the route we need," the second artifact is the one
that preserves the user's actual problem.

This tier therefore has to be used with an adversarial mindset:

- what does this artifact prove?
- what tempting story does it invite?
- what harder question does it still fail to close?

If that third question is skipped, retrieval degenerates into component
inventory with a confidence problem.

## Practical retrieval sequence for future doc work

When writing or revising a serious page, use this sequence:

1. name the exact claim category:
   - runtime
   - intent
   - plan
   - exploration
   - archive pressure
2. read the strongest source in that category first
3. pull one layer lower only if the stronger source is insufficient
4. capture both positive and negative evidence
5. explicitly note where the evidence ceiling stops

If a statement crosses categories, split it.
Do not use one sentence to smuggle:

- plan as runtime
- intent as proof
- exploration as current state

Also do not use one page structure to smuggle a smaller question in place of
the real one.

A page can be accurate at the sentence level and still misalign the repo if it
quietly changes:

- "how do requests survive the wrong node?" into "how is ingress set up?"
- "how do we remove sacred-node memory?" into "what components are present?"
- "what is the smallest honest middle layer?" into "what tools exist?"

This is the discipline the earlier docs were missing.

## Negative-source rules

These rules are here because this repo's retrieval failures are usually
category mistakes, not search failures.

- Do not let a planning document answer a runtime question unless you clearly
  mark it as plan-only.
- Do not let a side-path branch answer whether the priority root implementation
  already behaves that way.
- Do not let a healthy-looking component inventory answer whether a request is
  preserved under wrong-node or backend-loss pressure.
- Do not let the archive answer what is live.
- Do not let the runtime answer what the user is frustrated with if the runtime
  itself is one of the things failing to express that frustration.
- Do not let one smooth synthesis page outrank the more stubborn artifact that
  still contains the negative fact.

That last rule matters more than it sounds.
This knowledgebase will drift again the moment elegant synthesis is allowed to
erase the artifact that still says "the route disappears here" or "this truth
surface is still missing."

## The assimilation rules

These are the rules the rewrite should keep following.

## Rule 1: live implementation outranks polished prose for live-runtime claims

If a narrative file and the root implementation disagree about what is live
today, the implementation wins.

That is why:

- root Compose beats plan language for current runtime inventory
- active labels beat architectural aspiration for the live router surface
- absence in the worktree beats a confident description in an intent file

This is the main guard against documentation inflation.

## Rule 2: plans stay plans

Planning docs are authoritative about direction.
They are not authoritative about what operators can rely on today.

They can prove:

- a gap exists
- a failure mode is recognized
- a proposed fix is serious

They cannot prove:

- the fix is active
- the fix is validated
- the fix has earned broader claims

## Rule 3: archive pressure restores the real question, not the runtime answer

The archive matters because it keeps the user's actual complaint visible.

It does not matter because it can magically promote runtime truth.

Use it to recover:

- why the user rejects certain classes of answer
- why wrong-node preservation is central
- why fake HA language is intolerable

Do not use it to imply:

- the repo already solved those problems

## Rule 4: negative facts are first-class retrieval results

A mature retrieval pass must surface not only what exists, but what does not.

Examples:

- a live root `services.yaml` is not currently present
- certain route-generation claims are stronger than current proof
- some side-paths are exploratory and not promoted

If negative facts disappear, the docs start inflating instantly.

This is one of the most important retrieval rules in the repo because absence
here is often the only thing preventing wishful system stories.

## Rule 5: contradictions should survive assimilation

Do not resolve contradictions in prose unless the worktree or proof actually
resolved them.

Important contradictions that should remain visible include:

- Compose-first readability vs distributed truth ownership
- no-heavy-orchestrator intent vs growing helper-layer tax
- any-node entry aspiration vs incomplete wrong-node proof
- ingress continuity vs stateful correctness

If a rewrite makes these contradictions feel too tidy, it is probably lying by
smoothing.

That is especially true when a page starts sounding "complete."

In this repo, completeness often arrives by removing the most painful
contradictions from view.

## Bottom line

This page is the knowledgebase's anti-fog contract.

If a future rewrite wants to sound more certain, more elegant, or more complete
than the source stack supports, this page is the reason it should stop and
separate the truth layers again before writing.

That is the only way these docs can stay aligned with what the user is actually
asking for: not more prose, but more honest reconstruction of the real problem,
the real options, and the real limits of what the current worktree proves.
